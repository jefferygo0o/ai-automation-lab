# Deploying AI Automation Lab to Render

This guide explains how to deploy the lab on Render.com using the included `Dockerfile.render` and `render.yaml`.

## What you get

- A single Bun web service that runs both the REST/SSE API and serves the built React SPA.
- Render's persistent disk for `data/agents`, `data/sandboxes`, and the SQLite DB.
- Render auto-generated `LAB_MASTER_KEY` secret for encrypting the secrets vault.

## One-click setup (Blueprint)

1. Push the repo to GitHub (or GitLab).
2. In Render dashboard, click **New → Blueprint**.
3. Connect the repo. Render reads `render.yaml` and creates the service.
4. Render builds the Docker image, starts the service, and provisions a persistent disk mounted at `/var/data`.
5. Open the public URL — first visit, register an account, configure agent secrets.

## Manual setup (no Blueprint)

1. Push the repo to GitHub.
2. Render dashboard → **New → Web Service** → pick the repo.
3. Set:
   - **Runtime**: Docker
   - **Dockerfile path**: `Dockerfile.render`
   - **Docker context**: `.` (repo root)
4. Add environment variables from `render.yaml` (or copy the file).
5. Add a **Disk**:
   - Mount path: `/var/data`
   - Size: 1 GB (starter) — agents, sandboxes, and the SQLite DB all live here.
6. Deploy.

## Persistent storage

The lab stores everything under `/var/data/lab`:

| Path | Contents |
| --- | --- |
| `/var/data/lab/lab.db` | SQLite database (users, chats, runs, agents metadata, memory, secrets ciphertext, …) |
| `/var/data/lab/agents/<id>/` | Per-agent filesystem (system.md, persona.md, skills.md, tools.md, memory.md, config.json, skills/, memory/) |
| `/var/data/lab/sandboxes/<id>/` | Per-agent sandbox workdirs |

The disk is mounted as a Render **persistent disk** so it survives deploys and restarts.

## What does NOT persist

- The `secrets` vault ciphertext in SQLite persists (encrypted with `LAB_MASTER_KEY`).
- API keys for OpenAI, Anthropic, NVIDIA, etc. — users add these via the **Secrets** page in the UI after first deploy.
- The `LAB_MASTER_KEY` is auto-generated by Render and stored in its env store, so it persists across deploys. Don't change it or you'll lose access to existing encrypted secrets.

## Required env vars

| Var | Default | Purpose |
| --- | --- | --- |
| `PORT` | `10000` | Render expects this port |
| `NODE_ENV` | `production` | — |
| `LAB_PROJECT_ROOT` | `/app` | Project root in the image |
| `LAB_DIST` | `/app/backend/dist-spa` | Where the SPA bundle is copied to during build |
| `LAB_DATA_DIR` | `/var/data/lab` | Persistent root for agents, sandboxes, DB |
| `LAB_DB_PATH` | `/var/data/lab/lab.db` | SQLite file |
| `LAB_AGENTS_DIR` | `/var/data/lab/agents` | Per-agent filesystem root |
| `LAB_SANDBOX_ROOT` | `/var/data/lab/sandboxes` | Per-agent sandbox workdir root |
| `LAB_MASTER_KEY` | (auto-generated) | Encryption key for secrets vault |

## Cost

- **Starter plan** ($7/mo) is enough for low/medium traffic — 512 MB RAM, 0.5 CPU.
- **Standard plan** ($25/mo) recommended for serious use — 2 GB RAM, 1 CPU.
- Persistent disk: $0.25/GB/month (1 GB free on starter).

## Limitations vs running locally

- **No Docker sandbox backend** — only the `local` backend works. Per-agent sandboxes still get isolation but rely on `realpath`-jailed paths.
- **No WebGPU / FFmpeg tier-2 tools may be slower** — cloud CPUs vs local.
- **Cold starts** — first request after idle takes ~5-15s on starter plan (Bun spins up).
- **No persistent MCP servers** — they auto-reconnect on boot via `mcpManager.startAll()` but any local-only MCPs (e.g. filesystem-based ones) need different config.
- **Sandbox network** — the default `egress` permission lets agents make outbound HTTP; no inbound ports.

## Sandbox network egress

By default, agent sandboxes inherit the host's network. To lock down which hosts agents can reach, set per-agent config:

```json
{
  "sandbox": {
    "network": "egress",
    "allowHosts": ["api.openai.com", "integrate.api.nvidia.com"]
  }
}
```

Note: the `local` sandbox backend doesn't enforce allowHosts (only the `docker` backend does). To get network isolation, deploy with Docker available and change `LAB_SANDBOX_BACKEND=docker` (not currently supported — see roadmap).

## Troubleshooting

**Service won't start:** check Render logs. Common issue: `LAB_DIST` doesn't exist. Verify the build step completed by checking the build log for `frontend/dist/`.

**Database locked errors:** SQLite under WAL mode handles concurrent reads, but if you see "database is locked", the disk may have hit IOPS limits. Upgrade the disk or reduce write frequency.

**Agents can't reach LLM providers:** check that the agent's `apiKeySecret` resolves to a value. Add API keys via the **Secrets** page; they encrypt with `LAB_MASTER_KEY` and store in SQLite.

**MCP servers won't start:** the lab auto-starts MCP servers on boot. If a server fails, it's logged but doesn't block startup. Check `/api/mcp/servers` for status.

## Manual one-off commands

After deploy, you can SSH into the running service via Render dashboard → Shell:

```bash
ls /var/data/lab          # check persistent storage
cat /var/data/lab/lab.db  # inspect SQLite (use sqlite3 CLI)
env | grep LAB_           # check env vars
```
